# Accessibility (A11Y) Guide

## Target Conformance

**WCAG 2.2 Level AA**

All new UI patterns must meet WCAG 2.2 AA success criteria. Notable differences from 2.1 AA: focus appearance (2.4.11), dragging alternatives (2.5.7), target size minimum (2.5.8).

---

## Implemented Patterns

### Focus-Visible Rings

All interactive elements use Tailwind `focus-visible:ring-2 focus-visible:ring-prestige-gold` (or the equivalent `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-prestige-gold` where outline must be suppressed first). This ensures keyboard focus is visible against the dark navy surfaces without affecting mouse/touch users.

Applies to: Navbar links, CTA buttons, FAQ accordion buttons, mobile menu toggle.

### Accordion / aria-expanded (FAQ)

`src/routes/casino/+page.svelte` — Casino FAQ section.

Pattern used:

```html
<button
  type="button"
  aria-expanded={openFaqIndex === i}
  aria-controls="faq-panel-{i}"
  onclick={() => toggleFaq(i)}
>
  Question text
</button>

<div
  id="faq-panel-{i}"
  role="region"
  aria-labelledby="faq-btn-{i}"
  hidden={openFaqIndex !== i}
>
  Answer text
</div>
```

- Single-open behaviour: `openFaqIndex: number | null` ($state rune, Svelte 5).
- `hidden` attribute completely removes the panel from the accessibility tree when closed (preferred over `display: none` via class, since `hidden` is universally honoured by ATs without CSS).
- Chevron rotates 180° via `transition-transform` when open — conveying state visually without relying solely on `aria-expanded`.
- Keyboard: Enter/Space toggle is automatic via `<button>` semantics.

### Semantic `<button>` over `<div onclick>`

All interactive elements in the audited scope use native `<button type="button">` or `<a href>`. No `<div onclick>` patterns were found in the three files covered. This provides keyboard operability (Enter/Space), focus management, and AT role announcements without polyfilling.

## Pending Audits

### Color Contrast — Gold on Navy

`text-prestige-gold` (#D4943A) on `bg-navy-card` (#0D1629):

- **Large text (18pt+ or 14pt+ bold)**: likely passes AA (3:1 minimum).
- **Small text / body copy**: estimated contrast ratio ~3.8:1 — **may fail AA (4.5:1 required)**. Needs designer-verified contrast check with final rendered values.

Action required: run a colour contrast checker (e.g. WebAIM Contrast Checker) with the exact computed colours from the browser and confirm. If failing, designer follow-up needed to adjust either the gold or the background.

### Skip-to-Content Link

`src/routes/+layout.svelte` already includes a skip link that targets `#main-content` and becomes visible on keyboard focus. That satisfies the core WCAG 2.4.1 bypass-blocks requirement for the shared shell.

Follow-up still worth doing: verify the link is announced clearly in screen readers and that page-level layout changes never remove or obscure the `#main-content` target.

### Landmark Regions

Page structure should still be verified with a screen reader to confirm landmarks are well-scoped:
- `<nav aria-label="Main navigation">` — present in Navbar.
- `<main>` — present in `+layout.svelte` and wraps the page slot.
- `<footer>` — present in `SEOFooter.svelte`.

Keep checking for duplicate or unlabeled navigation regions when new shells or admin layouts are added.

## Running axe-core Locally

> Placeholder: Team E will wire Playwright + axe-core in a future sprint.

When the Playwright test suite is available, add axe accessibility checks as follows:

```typescript
// tests/a11y.test.ts (Playwright + axe-core)
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

test('casino page has no critical a11y violations', async ({ page }) => {
  await page.goto('/casino');
  const results = await new AxeBuilder({ page })
    .withTags(['wcag2a', 'wcag2aa', 'wcag22aa'])
    .analyze();
  expect(results.violations).toEqual([]);
});
```

To run axe-core in isolation today (without Playwright):

1. Install: `pnpm add -D axe-core` (dev dependency, not committed until test suite is set up).
2. In browser DevTools console on any page: paste the axe CDN snippet from [axe-core releases](https://github.com/dequelabs/axe-core/releases) and run `axe.run()`.
3. Review `violations` array in the console output.

Priority pages to audit: `/casino`, `/` (homepage), `/responsible-gambling`.
