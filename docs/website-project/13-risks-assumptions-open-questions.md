# 13 风险、假设与开放问题

## 风险

| 风险 | 影响 | 可能性 | 缓解措施 | 负责人 |
| --- | --- | --- | --- | --- |
| 编辑口吻与第一方口吻混用，容易让用户和审核方困惑 | 高 | 中 | 为不同页面族群统一口吻，并写进模板 | 产品 + 内容 |
| 缺少 `/home` 和 `/features/[slug]` 会破坏导航与 canonical | 高 | 高 | 创建缺失路由，或改为指向规范目标 | 工程 |
| 占位壳页面看起来像已完成内容 | 高 | 高 | 数据未准备好之前不要进入公开导航 | 工程 + 内容 |
| 合规敏感说法过期 | 高 | 中 | 按计划复审年龄、奖金、提现与法律文案 | 内容 + 法务 / 合规 |
| robots / AI map 的副本与真实路由树不同步 | 中 | 中 | 保持单一来源，并从它重新生成公开索引文件 | 工程 |
| 搜索索引或 sitemap 指向不存在的内容 | 中 | 中 | 路由变化前后都运行覆盖检查 | SEO + 工程 |

## 假设

| 假设 | 为什么重要 | 如何验证 |
| --- | --- | --- |
| 网站面向的是想看赌场、体育博彩和支付信息的加拿大成年人 | 决定口吻、导航与法律框架 | 查看利益相关者 brief 和批准目标 |
| 公开内容应对受监管说法保持谨慎 | 影响文案、转化与法务复核 | 上线前检查合规清单和页面文案 |
| 动态壳页面本来就要接 CMS 或数据源 | 决定页面该保留、隐藏还是补建 | 确认内容源与实现负责人 |
| Vercel 和 Node 24 仍是部署契约 | 影响运维与运行时决策 | 检查构建与部署设置 |

## 开放问题

| 问题 | 需要谁提供 | 优先级 | 是否阻塞 |
| --- | --- | --- | --- |
| `/home` 应该存在，还是所有引用都改成 `/`？ | 产品 + 工程 | 高 | 是 |
| 评测路径应继续放在 `/reviews/[slug]`，还是迁移到 `/features/[slug]`？ | 产品 + SEO + 工程 | 高 | 是 |
| 哪个 CMS 或数据源会驱动新闻、赛事、支付和指南壳页面？ | 内容 + 工程 | 高 | 是 |
| 哪些页面属于上线必需，哪些可以后续再做？ | 利益相关者 | 高 | 是 |
| 站点是否应该使用统一编辑口吻，还是部分页面继续保留第一方产品口吻？ | 利益相关者 + 内容 | 高 | 是 |
| 当前法律、年龄、提现和奖金说法是否都已确认可上线？ | 法务 / 合规 | 高 | 是 |
| `static/robots.txt` 是否要继续保留为与 live route 相同的副本？ | 工程 | 中 | 否 |
