# 09 给非工程人员看的技术概览

## 检测到的框架 / 平台

站点使用 **SvelteKit 2**、**Svelte 5**、**TypeScript**、**Tailwind CSS 4** 和 **Vercel**。应用配置为运行在 **Node 24.x**。

## 页面如何组织

- 每个页面都放在 `src/routes/` 下的路由文件夹里。
- 共享 UI 组件放在 `src/lib/components/`。
- 共享品牌与 SEO 值放在 `src/lib/site.ts` 和 `src/lib/ibet-brand.ts`。
- 样式分布在 `src/styles/`，再由 `src/app.css` 汇总。
- 公开静态文件放在 `static/`。

## 内容放在哪里

当前大部分公开内容都是直接写在 route 文件里。也就是说，页面正文、标题、元数据和结构化数据都属于应用的一部分。

代码库里也有一些页面壳，文案里说以后会接 CMS。但从这个仓库本身来看，还没有真正接好 CMS，所以这些页面要先当作占位页看待，直到被验证为止。

## 样式如何工作

- Tailwind 负责工具类。
- 设计 token 直接用 CSS 定义，而不是放在很大的 JavaScript 配置里。
- 站点用了自托管字体和模块化 CSS 层。
- 视觉风格是深色海军蓝配金色与蓝灰色点缀。

## 部署大致如何工作

站点面向 Vercel 部署。仓库里已经包含：

- `svelte.config.js` 中的 Vercel adapter
- 带 cron 计划的 `vercel.json`
- 基于路由的 API / workflow 端点
- 分析与速度洞察的挂载逻辑

## 哪些地方比较容易改

- 页面文案
- CTA 标签
- 图片和静态资源
- 导航标签
- 元数据
- 大多数 route 级内容
- 页脚链接

## 哪些地方改动风险较高

- 重定向与 canonical
- 年龄门禁和管理后台认证流程
- 公开法律 / 奖金 / 提现措辞
- workflow 端点与密钥
- 看起来像已完成、实际上还没接数据的路由壳页面

## 当前可见的技术债

- 真实的 `/robots.txt` 路由和静态 `static/robots.txt` 文件同时存在。
- 共享链接里有些路径还不存在，尤其是 `/home` 和 `/features/[slug]`。
- 一些动态页面仍然是占位壳，没有真实数据源。
- 有些页面文案还在引用错误的技术栈，或者品牌口吻不统一。
- 内容模型一部分像编辑，一部分像产品，这比单一风格更难维护。

## 建议的维护方式

- 内容页和 SEO 页要按计划复审，而不只是上线时检查一次。
- 高风险页面要先核实，再当作当前事实使用。
- 路由变化要同步更新 sitemap、搜索索引和文档。
- 内部占位页不要进入公开导航，直到完成。
- 把这套文档当成交付层，交给非工程人员使用。
